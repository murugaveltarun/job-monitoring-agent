# 2026-05-19 — Job Monitoring Agent

## Summary
Initial build of the enterprise job-monitoring agent, MLflow/UC registration debugging, pivot to Databricks Apps after Model Serving was blocked on the trial workspace, and first cut of a Teams alerting notebook. 14 commits total — see git log for full audit.

## Phase 1 — Initial build of the agent
Built a LangGraph ReAct agent that answers natural-language questions about Databricks job runs by querying a single `jobs.default.job_logs` Delta table.

- **Data layer** — [01_setup_table.py](../01_setup_table.py) creates and seeds `job_logs` with ~500 deterministic fake runs.
- **Package layer** ([job_monitor/](../job_monitor/)) — local-testable Python:
  - [job_monitor/config.py](../job_monitor/config.py): pydantic config loader.
  - [job_monitor/sql_guard.py](../job_monitor/sql_guard.py): sqlglot-based hard safety boundary — only single SELECT/WITH against the allowlisted table is accepted.
  - [job_monitor/tools.py](../job_monitor/tools.py): the `query_job_logs` tool, with both Spark (notebook) and SQL-warehouse (served) executors.
  - [job_monitor/agent.py](../job_monitor/agent.py): builds a LangGraph ReAct agent over the `databricks-meta-llama-3-3-70b-instruct` foundation model.
- **Interactive iteration** — [02_agent.py](../02_agent.py) notebook with an ~8-case eval harness.
- **Tests** — [tests/](../tests/) — local pytest suite, no Databricks needed.

## Phase 2 — MLflow → UC → Model Serving registration debug
[03_log_and_register.py](../03_log_and_register.py) logs [chain.py](../chain.py) to MLflow, registers it in Unity Catalog, then calls `agents.deploy(...)` to create a Model Serving endpoint. Took several iterations to get past UC and the deploy step.

- `e1afb91` Set `warehouse.http_path` in [config.yaml](../config.yaml) so the served-model path could authenticate against a SQL warehouse.
- `a0d274e` Defer warehouse env-var reads to query time — at `log_model` time `DATABRICKS_HOST/_TOKEN` aren't set, only at serving request time.
- `6d1b6ab` + `f8a7c7e` Pin `langgraph` + `langgraph-prebuilt` versions to fix an API-mismatch import failure.
- `b086d21` Register the model under `jobs.default.*` instead of `main.default.*` so the UC catalog matches the table's catalog.
- `1d384a6` Wrap the chain in a `RunnableLambda` that returns the final message's content as a string — `agents.deploy()` only accepts `StringResponse` or `ChatCompletionResponse` signatures.
- `e74b787` Declare an explicit MLflow signature via `infer_signature` so UC accepts the model version.

## Phase 3 — Blocker: Model Serving requires premium tier
Once `03_log_and_register.py` could log + register cleanly, `agents.deploy()` itself failed because Model Serving is a premium-tier feature and we're on a trial Azure Databricks workspace. The notebooks are otherwise correct — `03` would work unchanged on a paid workspace.

## Phase 4 — Pivot to Databricks Apps
Chose Databricks Apps as the alternative: it runs on dedicated Apps compute (works on trial), gives a real HTTPS endpoint, and the agent itself only needs the FM endpoint + SQL warehouse — neither of which require Serving.

### `ab59c7a` Add Databricks App front-end
- **New [app.py](../app.py)** — FastAPI app that builds the same agent as `chain.py` but exposes it as `POST /chat` over HTTP. Optional `X-App-Secret` header gate via env var.
- **New [app.yaml](../app.yaml)** — Apps manifest: uvicorn launch command + secret binding for `APP_SHARED_SECRET`.
- **Modified [requirements.txt](../requirements.txt)** — added `fastapi` + `uvicorn` (additive; doesn't break the serving notebook).

### `dc389b2` Fix warehouse auth for Apps
First deploy was successful but `/chat` returned 500 — Apps don't inject `DATABRICKS_TOKEN` (they use OAuth M2M via `DATABRICKS_CLIENT_ID` / `_SECRET`). Replaced direct env reads in [job_monitor/tools.py](../job_monitor/tools.py) with `databricks-sdk` `Config()`, which auto-detects whichever auth method is present. Same code now works in both Apps and Model Serving.

### Deployment verification
- Secret scope `job-monitor-app` created with key `shared_secret`.
- App `job-monitor-agent` created in Apps UI, resources attached: SQL warehouse, FM serving endpoint, and the secret aliased as `shared_secret` (matching `valueFrom` in [app.yaml](../app.yaml)).
- Deployed successfully — `/health` returned 200, `/chat` returned a real LLM answer: `{"reply": "There were 0 failed runs in the last 24 hours."}` (verified with a valid request body — the 500s before that were placeholder values like `"role": "string"` being sent literally).

## Phase 5 — Teams alerting (started)
### `d7cbead` Add scheduled-job alert notebook
- **New [04_alerts.py](../04_alerts.py)** — Databricks notebook that:
  - Queries [job_monitor/](../job_monitor/) table for new failures in the last `LOOKBACK_MINUTES` and any `RUNNING` jobs over `LONG_RUNNING_THRESHOLD_MINUTES`.
  - Bails silently if there's nothing to report.
  - Posts a MessageCard to a Teams Incoming Webhook URL stored as a Databricks secret (`teams_webhook_url` in scope `job-monitor-app`).

### Setup remaining (not yet done)
